# Title: 과제 구현

# Description: 승인된 설계안을 기반으로 작은 수직 조각을 구현한다.

1. `.agents/skills/lklass-conventions-check/scripts/refresh-rules.sh`를 실행하여 과제 규칙을 읽어라.
2. 승인된 `implementation_plan.md`를 읽어라. 승인된 설계안이 없으면 "먼저 `/design`을 실행해주세요."라고 안내하고 멈춰라.
3. `vertical-slicer` 스킬을 활성화하여 설계안을 구현 가능한 조각으로 쪼개라.
   - 조각은 가능하면 Controller-Service-Repository-Entity-Test-README 일부까지 관통하는 수직 조각으로 만든다.
   - 조각 목록을 유저에게 보여주고 승인받는다.
   - 승인된 조각을 `task.md`에 의존성 순서대로 기록한다.
4. 수직 구현 루프를 조각 하나당 반복한다.
   - 각 작업을 시작하기 전에 이번 작업에서 만들거나 수정할 파일/기능, 구현 방식, 필요한 이유, 검증 방법을 간단히 설명하고 유저 승인을 받아라.
   - 유저가 승인하기 전에는 파일 생성/수정으로 넘어가지 마라.
   - 한 번의 턴에 최대 5개의 파일만 생성/수정하는 것을 기본 원칙으로 하되, 이 제한은 리뷰 가능한 논리 단위를 유지하기 위한 장치다.
   - 파일 수 제한을 맞추기 위해 중복 구현, 부자연스러운 중첩 클래스, 과도하게 큰 파일, 나쁜 의존성 방향을 선택하지 마라.
   - 올바른 구조상 5개를 넘는 파일 수정이 필요하면 먼저 이유와 범위를 설명하고 유저 승인을 받아 진행하라.
   - 상태 변경은 의미 있는 도메인 메서드로 제한한다.
   - 정원 초과와 동시성 위험이 있는 작업은 테스트 가능한 방식으로 구현한다.
   - 테스트는 `given / when / then` 구획으로 작성하고, `@DisplayName`은 한국어로 구체적인 검증 의도를 드러내라.
   - 파일 작성 후 `lklass-conventions-check` 스킬로 점검한다.
   - 가능한 focused test를 실행한다.
   - 에러가 나면 `diagnose` 스킬의 재현 -> 가설 -> 계측 -> 수정 절차를 따른다.
   - 하나의 조각이 완료되면 커밋 메시지를 제안하고, 유저 승인 전에는 commit하지 않는다.
   - 유저에게 "다음 조각 진행할까?" 확인을 받는다.
5. 모든 조각이 완료되면 "모든 구현 조각이 완료되었습니다. `/test`로 테스트를 보강하거나 `/review`로 자가 검토를 진행하세요."라고 안내하라.
